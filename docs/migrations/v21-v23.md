# Migrate a v21 Brain to a Fresh v23 Brain

Use this procedure when migrating a user's existing v21 brain into a freshly initialized v23 brain.

This is an agent-executed migration runbook, not a deterministic script. Read the source brain carefully, make conservative decisions, and stop when the live behavior does not match the expected migration path.

## Goal

Move all user knowledge from a v21 brain into a fresh v23 brain while preserving:

- manual knowledge files
- synced source files
- source registration
- attachment syncing
- trusted summaries from the old `insights/` tree

The target outcome is:

- the v23 brain has the same user knowledge structure as v21
- imported summaries are accepted as baseline
- targeted regen checks do not rewrite imported summaries

## Preconditions

Before beginning, verify all of the following.

1. The target brain is freshly initialized with `brain-sync init`.

Expected minimum structure:

```text
<v23-root>/
  .brain-sync/brain.json
  knowledge/
```

2. The user's source auth is already configured for both Confluence and Google Docs.

Check `C:\Users\<user>\.brain-sync\config.json`.

Verify:

- `confluence.domain`
- `confluence.email`
- `confluence.token`
- `google.token`

If either auth block is missing or clearly incomplete, stop before migration.

3. Treat the v21 brain as read-only.

Do not migrate the old runtime DB.
Do not migrate top-level `insights/`, `schemas/`, `.sync-state.sqlite`, or `.brain-sync/version.json` as-is.

4. The daemon should be stopped while the migration is in progress.

This avoids mixed watcher/sync/reconcile activity while files and manifests are being moved into place.

## Migration Rules

- Copy knowledge content, not the old runtime state.
- Do not manually copy legacy `_attachments/` directories.
  Let v23 rehydrate attachments by syncing registered sources with `--sync-attachments`.
- Do copy old `summary.md` files from `v21/insights/...` into the new co-located v23 layout.
- If a v21 knowledge area has no summary, still migrate the knowledge and source registration, but record that the area will require regen later.
- It is acceptable to skip source children during this migration.
  Register only the files materially present in the v21 knowledge tree.

## Step 1: Inventory the v21 Brain

Start by inventorying:

- top-level knowledge folders under `v21/knowledge`
- synced files
- manual files
- old summaries under `v21/insights`

Classify a markdown file as synced if its header contains a legacy source marker like:

```html
<!-- brain-sync-source: confluence:1234567890 -->
```

or:

```html
<!-- brain-sync-source: gdoc:documentId -->
```

If the header is absent, treat the file as manual content.

## Step 2: Copy User Knowledge

Copy each v21 knowledge area into the v23 `knowledge/` tree.

Exclude legacy managed/runtime artifacts:

- `_attachments`
- `.brain-sync`
- `insights`
- `schemas`
- `_sync-context`

Keep all user-visible files and folders, including manual content.

This copy should preserve the user's knowledge structure, subject only to any deliberate v23 normalization already agreed for that brain.

## Step 3: Register Synced Sources

For each synced file discovered in v21, register a v23 source manifest.

Always enable attachments on add:

```powershell
brain-sync add "<url>" --path "<knowledge-path>" --sync-attachments --root <v23-root>
```

URL rules:

- Confluence:
  `https://<domain>/wiki/spaces/unknown/pages/<pageId>`
- Google Docs:
  `https://docs.google.com/document/d/<docId>/edit`

Use the source identity from the file header to derive the canonical source:

- `confluence:<pageId>`
- `gdoc:<docId>`

Use the containing knowledge folder as the `--path`.

Examples:

```powershell
brain-sync add "https://serko.atlassian.net/wiki/spaces/unknown/pages/5381161064" --path "initiatives/B4B - Native Flight Experience/L2 AERD B4B - Flights Search (R2)" --sync-attachments --root C:\path\to\v23
```

```powershell
brain-sync add "https://docs.google.com/document/d/10Lmqq6OYBw2bn6GCUllYwJ5d-q4gQh1h4S1SZEf63pY/edit" --path "initiatives/B4B - Native Flight Experience/PRD Native Flight Experience" --sync-attachments --root C:\path\to\v23
```

Do not use child-sync features in this migration.

## Step 4: Copy Old Summaries Into the v23 Layout

Map each old summary:

```text
v21/insights/<area>/summary.md
```

to:

```text
v23/knowledge/<area>/.brain-sync/insights/summary.md
```

Important mappings:

- old root:
  `v21/insights/summary.md`
  ->
  `v23/knowledge/.brain-sync/insights/summary.md`
- old `_core`:
  `v21/insights/_core/summary.md`
  ->
  `v23/knowledge/_core/.brain-sync/insights/summary.md`

Copy summaries at every level that exists in v21 and has a corresponding area in v23.

If a summary is missing in v21:

- still migrate the knowledge area
- still register its sources
- note the missing summary explicitly
- expect regen for that area later

## Step 5: Adopt Imported Summaries as Baseline

Run:

```powershell
brain-sync doctor --root <v23-root> --adopt-baseline
```

This records the current v23 summaries as the accepted baseline.

### Important Existing-State Rule

`--adopt-baseline` will not overwrite an area that is already baselined.

If you need to replace an already-baselined v23 summary with the old v21 summary:

1. Copy the v21 `summary.md` over the v23 `summary.md`.
2. Delete that area's:

```text
<area>/.brain-sync/insights/insight-state.json
```

3. Re-run:

```powershell
brain-sync doctor --root <v23-root> --adopt-baseline
```

Do this only for the specific areas whose baseline you are replacing.

## Step 6: Run Targeted Regen Acceptance Checks

After baseline adoption, verify imported summaries are trusted.

Use targeted regen checks on representative imported paths:

```powershell
brain-sync regen --root <v23-root> "<knowledge-path>"
```

Expected good result:

```text
Done. 0 insight files regenerated.
```

This is the preferred acceptance signal.

### Stop Conditions

Stop the migration and investigate if any imported area does one of the following during a targeted acceptance check:

- rewrites `summary.md`
- reports `1` or more regenerated insight files
- unexpectedly calls model generation for an area that should already be baselined

If the output says the model generated a candidate summary but discarded it as `>97% similar`, that usually means the area was not fully accepted as baseline yet. Fix the baseline state before continuing.

### Important Root Warning

Do not use:

```powershell
brain-sync regen --root <v23-root>
```

as a "root only" check.

With no path, this runs regen for the whole brain.

If you need to replace the top-level root summary:

- copy the old root summary into `knowledge/.brain-sync/insights/summary.md`
- clear `knowledge/.brain-sync/insights/insight-state.json`
- re-run `doctor --adopt-baseline`

## Step 7: Start the Daemon and Watch Startup Reconcile

Once the content, manifests, and baseline summaries are in place, start:

```powershell
brain-sync run --root <v23-root>
```

Watch startup logs carefully.

Expected behavior:

- source sync should bind registered manifests to copied files
- attachments should begin rehydrating for synced sources
- startup reconcile should detect offline-imported knowledge areas cleanly
- imported, baselined areas should not trigger a large avoidable regen wave

If unexpected regen starts rewriting imported summaries, stop and investigate before continuing.

### First Live Sync Reality

The first daemon startup may still update some synced files even after a careful import.

Common reasons:

- legacy v21 managed headers are rewritten into v23 frontmatter
- legacy attachment links such as `_sync-context/attachments/...` are rewritten into `.brain-sync/attachments/...`
- remote source content has changed since the v21 brain snapshot was taken

This can trigger a one-time regen wave on imported synced areas. If that happens:

1. Let the first sync complete.
2. Stop the daemon.
3. Re-copy the old v21 summaries for the affected areas if needed.
4. Clear the affected `insight-state.json` files.
5. Re-run:

```powershell
brain-sync doctor --root <v23-root> --adopt-baseline
```

6. Re-run targeted regen acceptance checks on those areas.

Do not continue until the targeted regen checks return:

```text
Done. 0 insight files regenerated.
```

Treat this post-first-sync re-baseline as the stabilization step that converts the freshly normalized synced files into the final accepted v23 baseline.

## Step 8: Validate the Final User Knowledge Structure

After migration, compare `v21/knowledge` to `v23/knowledge`, ignoring managed/runtime artifacts.

Ignore:

- `.brain-sync`
- `_attachments`
- `insights`
- `schemas`
- `_sync-context`

Validate:

- the same user knowledge directories exist
- the same user knowledge files exist
- any intentional normalization is understood and documented

Example validation approach:

```python
from pathlib import Path

EXCLUDED_DIRS = {".brain-sync", "_attachments", "insights", "schemas", "_sync-context"}

def collect(root: Path):
    result = set()
    for path in root.rglob("*"):
        rel = path.relative_to(root)
        if any(part in EXCLUDED_DIRS for part in rel.parts):
            continue
        result.add((path.is_dir(), rel.as_posix()))
    return result

v21 = collect(Path(r"C:\path\to\v21\knowledge"))
v23 = collect(Path(r"C:\path\to\v23\knowledge"))

print("Only in v21:", sorted(v21 - v23))
print("Only in v23:", sorted(v23 - v21))
```

The expected result is either:

- no differences, or
- only deliberate, documented normalizations

## Expected Outcome

A successful migration leaves the v23 brain with:

- the full user knowledge structure from v21
- synced sources re-registered
- attachment sync enabled
- old summaries converted into the co-located v23 layout
- imported summaries trusted as baseline
- no unexpected regen after targeted acceptance checks

## If Something Is Missing

If a v21 area is missing its old summary:

- migrate the knowledge anyway
- migrate source registration anyway
- record the missing summary explicitly
- treat that area as a planned regen exception

This should be unusual, but it must not block the broader migration.
