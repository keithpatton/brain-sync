# brain-sync 0.5.0 -> 0.7.0.0 Guided Migration

This guide describes the approved guided migration for existing
`brain-sync 0.5.0` brains onto the current `brain-sync 0.7.0.0` contract.

Use this migration when an existing brain still carries the older Brain Format
`1.0` source-manifest contract with:

- `materialized_path`
- `target_path`
- `status`
- `sync_hint`

The guided migration target is:

- app version `0.7.0.0`
- Brain Format `1.2`
- runtime DB schema `v29`

---

## What Changes

The major portable source-state shift still happens at the `1.0 -> 1.1`
boundary. The guided `1.0 -> 1.2` migration therefore applies that same
manifest remap and then lands on the final `1.2` portable contract.

Brain Format `1.1` replaces the old source lifecycle split with:

- `knowledge_path`
- `knowledge_state`

It also moves the portable freshness token to:

- `remote_fingerprint`

Runtime ownership narrows at the same time:

- runtime DB schema becomes `v26`
- runtime source progress table becomes `sync_polling`
- portable freshness and materialization facts no longer live in runtime DB

Brain Format `1.2` then finalizes the portable boundary by:

- keeping `knowledge_path` and `knowledge_state` as the durable portable
lifecycle contract
- keeping `content_hash`, `remote_fingerprint`, and `materialized_utc` as the
last-successful portable materialization baseline
- removing portable `missing_since_utc`
- leaving missing-observation timing and explicit-finalization coordination in
machine-local runtime state instead of the portable manifest

---

## Manifest Mapping

Map each Brain Format `1.0` source manifest into the Brain Format `1.2`
contract as follows.

### Path mapping

`knowledge_path` is determined in this order:

1. if `materialized_path` is non-empty, use `materialized_path`
2. otherwise derive a provisional anchored path from `target_path` plus the
  source-dir ID filename

Examples:

- `target_path = "teams/platform"` and `canonical_id = "confluence:12345"`
becomes `knowledge_path = "teams/platform/c12345.md"`
- empty `target_path` with `canonical_id = "test:alpha"` becomes
`knowledge_path = "alpha.md"`

### Lifecycle mapping


| Brain Format 1.0                                       | Brain Format 1.2                   |
| ------------------------------------------------------ | ---------------------------------- |
| `status = "active"` and no materialized success fields | `knowledge_state = "awaiting"`     |
| `status = "active"` and a settled materialized file    | `knowledge_state = "materialized"` |
| `status = "missing"`                                   | `knowledge_state = "missing"`      |


There is no direct Brain Format `1.0` value for `stale`; it is written by
Brain Format `1.2` workflows when a moved or rediscovered file must be
rematerialized.

### Freshness mapping

If `sync_hint` exists:

- `sync_hint.content_hash` -> `content_hash`
- adapter-owned prior freshness token, when available -> `remote_fingerprint`

If the old manifest has no durable remote token, leave `remote_fingerprint`
unset and require the next successful materialization to establish it.

### Time mapping

If the old manifest has a last-successful sync timestamp, map it to
`materialized_utc`.

If `status = "missing"`, preserve the missing lifecycle state itself by writing
`knowledge_state = "missing"`.

Do not keep portable `missing_since_utc` in the final Brain Format `1.2`
manifest. That field is retired in `1.2`. Missing-observation timing belongs
to machine-local runtime state after upgrade.

---

## Validity Rules After Migration

The migrated manifest must satisfy the Brain Format `1.2` state matrix:

- `awaiting`: no `content_hash`, `remote_fingerprint`, or `materialized_utc`
- `materialized`: `content_hash`, `remote_fingerprint`, and `materialized_utc`
set
- `stale`: same last-successful fields as `materialized`
- `missing`: no portable missing timestamp field; last-successful fields may
remain

Invalid mixed old/new manifests should be corrected during migration rather
than preserved.

## Runtime DB Migration

Runtime DB migration is deterministic and separate from the portable migration.

For the supported `0.5.0 -> 0.7.0.0` path, the runtime DB moves from `v23` to
`v29` through the supported in-place schema steps:

- `v23 -> v24`
- `v24 -> v25`
- `v25 -> v26`
- `v26 -> v27`
- `v27 -> v29`

The `v25 -> v26` step:

1. creates `sync_polling`
2. copies polling/scheduling columns from `sync_cache`
3. drops `sync_cache`
4. writes `schema_version = 26`

The `v26 -> v27` step adds the machine-local `source_lifecycle_runtime` table
used for missing-observation history and source-level lifecycle leases.

The `v27 -> v29` step removes the unreleased same-session confirmation fields
that are not part of the shipped `0.7.0.0` contract while retaining only the
machine-local missing-observation timestamps and lease fields.

If an unreleased provisional `v28` runtime DB is encountered, it must also be
migrated forward to `v29` before use. It is not a released compatibility row.

Portable source truth is not reconstructed from runtime DB rows during this
step.

---

## Post-Migration Expectations

After migration:

- registration and reconcile decisions are recoverable from the portable
manifest alone
- a moved or rediscovered file is written back as `knowledge_state = stale`
immediately
- `stale` must force a full rematerialization even when the
`remote_fingerprint` matches
- `missing` remains portable and durable, but missing-observation timestamps no
longer live in the portable manifest
- deleting or rebuilding runtime DB state must not alter portable source truth
