## New Analysis of plan_v23_2

### Architectural Assessment: Strong

The plan correctly resolves every ambiguity from review 1. The four
atomic cutovers (A-D) are the most important structural addition — they
prevent the Phase boundary violations that were the primary risk in the
original plan. The cutover model is the right abstraction for guiding
implementation agents through a refactor of this scope.

### Remaining Issues

After thorough verification against the codebase, I found two issues
worth addressing and one advisory note. None are blocking.

---

#### I1. Phase 1 transitional DB schema must retain `documents` and `relationships` tables (Minor, Clarify)

Phase 1 says "stamp the runtime DB schema integer as `23`" and "delete
the v21 migration chain." Cutover A says don't drop `documents` and
`relationships` until Phase 4.

The implementation agent will need to write a fresh `CREATE TABLE`
block for the Phase 1 schema. The v23 spec (SCHEMAS.md) defines only
4 tables. But Cutover A requires the tables to exist until Phase 4.

**Ambiguity:** When the agent writes the Phase 1 DDL, should they
include `documents`, `relationships`, and their indexes as transitional
tables in a schema stamped as `23`? Or should they keep schema version
at `21` until Phase 4?

**Recommendation:** Add a one-line clarification to Phase 1: "The Phase
1 DDL includes `documents` and `relationships` as transitional tables.
Phase 4 removes them and finalizes the v23 4-table schema. The schema
integer remains `23` throughout — the transitional tables are permitted
as implementation scaffolding."

This is minor because the cutover already implies it, but making it
explicit prevents the agent from wondering whether to include or exclude
these tables during Phase 1.

#### I2. `state.py` should be listed in Phase 2 primary modules (Minor, Omission)

Phase 2's implementation requirements say "remove `fetch_children` and
`child_path` from: `SourceManifest`, `SourceState`, daemon scheduling/
state merge logic." `SourceState` is defined in `state.py` at line 212.
`load_state()` in `state.py` creates `SourceState` objects and
populates `fetch_children` from manifests.

But Phase 2's primary modules list does not include `state.py`:

```
- src/brain_sync/manifest.py
- src/brain_sync/pipeline.py
- src/brain_sync/commands/sources.py
- src/brain_sync/cli/__init__.py
- src/brain_sync/cli/handlers.py
- src/brain_sync/sources/__init__.py
- src/brain_sync/fileops.py
- src/brain_sync/mcp.py
```

**Recommendation:** Add `src/brain_sync/state.py` to Phase 2's primary
modules list. The agent needs to modify `SourceState`, `load_state()`,
`_seed_from_hint()`, and `clear_children_flag()` in this phase.

#### I3. plan_v23_supporting.md says Phase 1 removes `fetch_children`/`child_path` from state helpers; plan_v23_2 defers to Phase 2 (Advisory)

plan_v23_supporting.md Phase 1 `state.py` row says: "Remove all references to
`fetch_children`, `child_path`, and `manifest_version` from state
helpers."

plan_v23_2 Phase 2 (via Cutover B) defers this to Phase 2.

The plan provides a mapping table, but it doesn't flag this specific
divergence. An implementation agent reading plan_v23_supporting.md Phase 1
alongside plan_v23_2 Phase 1 might do the cleanup early, breaking
`pipeline.py` and `__main__.py` before Cutover B is complete.

**Recommendation:** Either:
- Add a note to the mapping table: "plan_v23_supporting.md Phase 1 includes
  `fetch_children`/`child_path` removal from state.py; plan_v23_2
  defers this to Phase 2 via Cutover B."
- Or update plan_v23_supporting.md Phase 1 to remove this item (since the
  plan is authoritative).

This is advisory because the plan explicitly states "this document
supersedes `PLAN_v23.md`" and the cutover mechanism should prevent
agents from landing partial changes. But clearing the contradiction
reduces cognitive load.

### What the plan gets right

1. **Atomic cutovers are the right model.** The four cutovers (A-D)
   correctly identify the dependency pairs that review 1 flagged. An
   implementation agent can check each cutover as a pre-commit gate.

2. **Phase 0 harness work is correctly scoped.** Including the E2E
   invariant checker, daemon readiness, and config isolation in Phase 0
   prevents the cascade failure that would otherwise start in Phase 1.

3. **Phase 4 isolating attachment + table drop is correct.** This was
   the most dangerous dependency in the original plan. Making it a
   single-slice cutover eliminates the intermediate breakage.

4. **Serialization guidance for Phase 3 is correct.** The "should NOT
   proceed in parallel" note for the co-located path cutover matches
   the review 1 finding about Phase 3 parallelism risk.

5. **`google_doc` durable value with internal mapping is pragmatic.**
   This avoids a risky end-to-end rename while ensuring spec compliance
   at the durable boundary.

6. **Command-path-only child discovery is the cleanest option.**
   Eliminating the deferred daemon-loop flag removes a fragile
   stateful flow.

7. **`_collect_global_context_structured()` in mcp.py** is implicitly
   covered. Phase 3 lists `mcp.py` in primary modules and says "replace
   top-level `insights/` path logic throughout regen and query
   surfaces." The MCP function reads from `schemas/` and
   `insights/_core/` — both of which Phase 3 eliminates. An agent
   updating `mcp.py` in Phase 3 will naturally find and update this
   function.

8. **Test volume estimate is realistic.** The 60-100 rewrite range
   aligns with the codebase analysis from review 1.
