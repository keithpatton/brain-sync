# PLAN_v23_3: Brain Format 1.0 / Runtime v23 Refactor

## Status

Planning artifact only. This document supersedes `plan_v23_2_2026-03-15.md` as the
implementation plan for aligning the codebase to the authoritative Brain
Format `1.0` and runtime DB schema `v23` documents under `docs/brain-format/`.

This stage does **not** change product behavior. It resolves several planning
ambiguities discovered during review and repo validation so the later
implementation stage can execute without guessing.

## Objective

Refactor the codebase so the supported baseline is exactly:

- app version `0.5.0`
- Brain Format `1.0`
- runtime DB schema `v23`

The implementation must follow the `docs/brain-format/` spec set when current
code conflicts with it. Pre-v23 layouts are unsupported legacy inputs for this
release and must not be treated as supported durable layouts.

## Authoritative inputs

The implementation must treat these documents as the source of truth:

- `docs/brain-format/RULES.md`
- `docs/brain-format/SCHEMAS.md`
- `docs/brain-format/VERSIONING.md`
- `docs/brain-format/COMPATIBILITY.md`
- `docs/brain-format/GLOSSARY.md`
- `docs/brain-format/TEMPLATES.md`

`docs/plans/plan_v23_supporting.md` is implementation guidance and should stay
aligned, but the spec set above is authoritative for behavior and acceptance.

## Scope

In scope for the implementation stage:

- filesystem contract alignment to Brain Format `1.0`
- source manifest schema and identity rule alignment
- materialization/frontmatter rule alignment
- co-located per-area `.brain-sync/` managed-state layout
- runtime DB relocation and schema reset to `v23`
- watcher, reconciliation, doctor, CLI, MCP, and query/path updates caused by
  the new layout and version contract
- test-harness and compatibility-suite work needed to prove the single
  supported baseline

## Non-goals

- no product code changes during this planning stage
- no migration or upgrade path from pre-v23 brains
- no attempt to preserve v21 durable layouts as supported inputs
- no new feature work beyond spec alignment
- no dependency on `brain-example/` as a test-fixture source

## Validated review outcomes

The external review surfaced useful issues. I validated them against the
current repository before incorporating them here.

Accepted as real planning gaps:

- DB schema integer ambiguity between `v23` and `1`
- durable `source_type` ambiguity between `googledocs` and `google_doc`
- incomplete planning for the `fetch_children` / `child_path` flow
- missing explicit decision on `_core` global-context inputs
- missing harness/test dependencies around daemon readiness and E2E invariants
- table-removal timing conflicts with current attachment implementation
- underestimation of test rewrite scope

Rejected after validation:

- `context.py`, `context_index.py`, and `link_rewriter.py` do not exist as
  source modules in `src/brain_sync/`; only stale compiled `.pyc` artifacts are
  present. They should not be added to the source-module impact list. Agents
  should audit actual `.py` sources, not compiled leftovers.

## Fixed decisions for implementation

These are now explicit and are **not** open questions:

- root durable manifest is `.brain-sync/brain.json` with `{"version": 1}`
- source manifests live at `.brain-sync/sources/<source_dir_id>.json`
- durable manifest version field name is `version`, not `manifest_version`
- runtime DB path is `~/.brain-sync/db/brain-sync.sqlite`
- runtime daemon status path is `~/.brain-sync/daemon.json`
- runtime DB integer schema constant is `23`
  - this is a clean-break schema with no migration chain, but it is still
    stamped as `23` to match the project baseline and compatibility row
- per-area managed state is co-located under `knowledge/<area>/.brain-sync/`
- the root area is valid and uses `knowledge/.brain-sync/...`
- `.brain-sync/` is the single reserved managed namespace and must be excluded
  consistently from readable-content discovery, hashing, regen, reconcile,
  watcher, and query assembly
- durable/on-disk Google Docs provider values use `google_doc` per spec
  - manifests, frontmatter, doctor output, and compatibility assertions must
    use `google_doc`
  - internal code may temporarily keep `SourceType.GOOGLE_DOCS` naming if a
    mapping layer is introduced at the boundary in the same implementation
    slice
- global context in v23 includes only raw files from `knowledge/_core/`
  - it does **not** include `schemas/`
  - it does **not** include `_core`'s generated summary
  - this is an intentional spec-alignment behavior change from current code
  - this special treatment applies to regeneration/global-context assembly,
    not deterministic search
  - for deterministic query/search, `_core` behaves like any other area via
    its generated meaning and other indexed artifacts
  - explicit context-loading surfaces may still expose raw `knowledge/_core/`
    when the caller asks for global context directly
- `fetch_children` and `child_path` become command-path-only parameters
  - they are not persisted in manifests
  - they are not carried in `SourceState`
  - they are not read by the daemon loop
  - one-shot child discovery happens in the same `add` or explicit `sync`
    invocation that requested it
- legacy runtime DBs are never migrated forward
  - a DB at the new runtime path with the wrong schema version is discarded and
    recreated because runtime state is rebuildable
  - a legacy brain-root `.sync-state.sqlite` is ignored by runtime startup and
    reported by doctor as unsupported legacy/runtime drift
- skill deployment to `~/.claude/skills/brain-sync/` is unchanged by this
  refactor

## Planning outcomes required before broad implementation

Before broad behavior rewrites, the implementation stage should establish:

1. Shared path/version helpers so layout and version rewrites are centralized.
2. Version-aware test builders and assertions so the suite stops encoding v21
   assumptions implicitly.
3. A dedicated `tests/compat/` suite for format/runtime compatibility
   contracts.
4. E2E harness updates that prevent every daemon/E2E test from failing as soon
   as the runtime path contract changes.

These four items are the foundation for the rest of the refactor.

## Atomic cutovers

The repo has several dependency pairs that should not be landed half-done.
Implementation agents should treat these as atomic cutovers even if the work is
organized under separate high-level phases.

### Cutover A: Runtime DB schema reset plus attachment consumer rewrite

Do not drop `documents` and `relationships` from the runtime DB while
`attachments.py` and `commands/sources.py` still call:

- `save_document()`
- `save_relationship()`
- `load_relationships_for_primary()`
- `remove_relationship()`
- `remove_document_if_orphaned()`

The table removal and filesystem-only attachment lifecycle rewrite must land in
the same slice.

### Cutover B: Manifest schema cleanup plus `SourceState` / daemon / pipeline cleanup

Do not remove persisted `fetch_children` and `child_path` from manifests unless
the same slice also:

- removes `SourceState.fetch_children`
- removes `SourceState.child_path`
- updates `process_source()` to take explicit function parameters only
- removes daemon-loop child-discovery flow from `__main__.py`
- updates CLI and MCP command paths that request child discovery

### Cutover C: Co-located `.brain-sync/` layout plus invariant helpers

Do not switch area paths to `knowledge/**/.brain-sync/` unless the same slice
also updates:

- shared path helpers
- E2E invariant assertions
- E2E daemon readiness helpers if they depend on old DB state
- root-area handling helpers
- summary and insight-state lookup helpers

### Cutover D: Managed-namespace exclusion at directory-walk level

Do not rely on file-level hidden-file checks alone. Every directory walk using
`iterdir()` or `rglob()` must skip `.brain-sync/` explicitly, or regen loops
and false content hashing will occur.

### Phase 1: Runtime baseline and init contract

Goals:

- move runtime ownership out of the brain root
- establish the Brain Format `1.0` init baseline
- remove v21 bootstrap artifacts from init paths

Implementation requirements:

- move DB ownership to `config.py` and resolve DB path as
  `~/.brain-sync/db/brain-sync.sqlite`
- stamp the runtime DB schema integer as `23`
- delete the v21 migration chain from `state.py`
- define explicit behavior for wrong-version runtime DBs at the new path:
  discard and recreate
- the Phase 1 DDL may temporarily retain `documents` and `relationships` as
  transitional tables while Cutover A is still open
  - the schema integer remains `23`
  - Phase 4 removes those tables and finalizes the 4-table v23 runtime schema
- replace DB `daemon_status` usage with `~/.brain-sync/daemon.json`
- ensure `brain-sync init` creates only:
  - `.brain-sync/brain.json`
  - `.brain-sync/sources/`
  - `knowledge/`
  - `knowledge/_core/`
- ensure `brain-sync init` does **not** create:
  - `insights/`
  - `schemas/`
  - `.sync-state.sqlite`
  - `.gitignore`
  - `knowledge/.brain-sync/`
- replace `.brain-sync/version.json` with `.brain-sync/brain.json`
  - no read tolerance is required for `version.json`; pre-v23 layouts are
    unsupported
- remove schema deployment from init in this phase, not later
- remove `.gitignore` creation from init in this phase, not later
  - existing user-owned `.gitignore` files must not be deleted or rewritten

Primary modules:

- `src/brain_sync/config.py`
- `src/brain_sync/state.py`

## Dependency order

Must happen first:

- foundation helpers
- E2E invariant checker rewrite
- daemon readiness harness rewrite
- compatibility test skeleton

Then execute in this order:

1. runtime/init/version baseline
2. manifest/frontmatter/child-discovery contract
3. co-located managed-state layout and template/context cleanup
4. attachment isolation plus DB table drop
5. watcher/reconcile/doctor/path consumers
6. compatibility closeout and full test rewrite

Can proceed in parallel after the foundation is in place:

- runtime/init/state work
- manifest/frontmatter/source-command work
- pure path-helper groundwork

Should **not** proceed in parallel:

- the co-located path cutover across regen, reconcile, watcher, doctor, and
  MCP consumers
- the attachment rewrite and DB table drop

## Test strategy

### Foundation work required early

- add version-aware builders:
  - `build_brain_v1(...)`
  - `build_drifted_brain_v1(...)`
  - `build_legacy_pre_v1_brain(...)`
- add compatibility assertions:
  - `assert_brain_format_v1(...)`
  - `assert_runtime_rebuildable(...)`
  - `assert_unsupported_legacy_brain(...)`
- rewrite E2E invariant checks before Phase 1 lands
- rewrite daemon readiness detection before `daemon_status` is removed

### Compatibility coverage required

- fresh init produces a conforming Brain Format `1.0` brain
- drifted supported-format brains can be repaired without loss of user
  knowledge
- runtime DB deletion and rebuild does not invalidate a valid brain
- unsupported pre-v23 layouts are detected and not silently treated as
  supported

### Feature-specific coverage required

- `brain.json` creation and repair
- legacy `version.json` rejection/reporting
- source manifest filename validation
- YAML frontmatter merge behavior
- frontmatter read fallback for legacy HTML comments
- durable `google_doc` writes
- command-path child discovery without persisted manifest flags
- root-area summary and insight-state behavior
- watcher handling for file moves, folder moves, and deletes
- reconciliation path repair via three-tier identity resolution
- attachment lifecycle under `.brain-sync/attachments/<source_dir_id>/`
- doctor checks and fix behavior for the v23 contract
- MCP/query path updates for new summary locations and global-context sources

### Test volume estimate

This is a high-churn refactor for the test suite.

Expected order of magnitude:

- rewrite: roughly 60-100 tests and several shared harness helpers
- remove or replace: roughly 15-30 v21-only tests
- new v23-specific tests: roughly 20-35

These are lower-bound planning numbers, not a hard cap. `test_regen.py`,
`tests/mcp/test_mcp.py`, the E2E harness, and doctor/integration suites are
likely the highest-churn areas.

## Acceptance criteria

Implementation is complete when all of the following are true:

- `brain-sync 0.5.0` creates and operates on Brain Format `1.0` brains with
  runtime DB schema integer `23`
- fresh init produces only the spec-approved baseline artifacts
- durable manifests and materialized-file identity match the v23 schema and
  frontmatter rules
- durable Google Docs writes use `google_doc`
- child discovery works through command paths without persisted manifest flags
- per-area managed state is co-located under `knowledge/**/.brain-sync/`
- global context is assembled from package instructions plus raw
  `knowledge/_core/` files only
- deterministic query/search does not get automatic raw `_core` injection;
  `_core` participates there as a normal area via generated meaning
- runtime DB and daemon state are fully outside the brain root
- deleting runtime DB state does not invalidate a valid brain
- wrong-version runtime DBs at the new location are safely recreated
- doctor validates supported brains, repairs repairable drift, and clearly
  rejects unsupported legacy layouts without attempting an upgrade
- compatibility tests prove the single supported row and pass without relying
  on `brain-example/`

## Risks and mitigations

- Risk: v21 assumptions are spread through code and tests.
  Mitigation: centralize new path helpers and rewrite high-churn harness code
  first.

- Risk: `.brain-sync/` content is accidentally treated as readable input.
  Mitigation: enforce one exclusion rule reused by discovery, hashing, regen,
  reconcile, watcher, query assembly, and tests at the directory-walk level.

- Risk: attachment logic still depends on removed DB tables.
  Mitigation: land attachment isolation and table removal as one atomic cutover.

- Risk: version drift between docs and code.
  Mitigation: treat app version, Brain Format version, runtime DB schema label,
  and runtime DB schema integer as explicit acceptance criteria and test
  assertions.

- Risk: leftover migration logic silently preserves legacy layouts.
  Mitigation: delete active migration flows, ignore legacy brain-root runtime
  DBs, and add explicit unsupported-legacy tests plus doctor messaging.

- Risk: Phase 3 path work is parallelized too aggressively.
  Mitigation: centralize path helpers first, then serialize the broad consumer
  rewrite across regen, reconcile, watcher, doctor, and MCP.

## Open questions kept for implementation

- Should the internal enum/string value for Google Docs also be renamed
  end-to-end after the durable boundary mapping lands?
  Default: not required for the first spec-alignment cutover if durable writes
  are correct and tests cover the boundary.

- Should package template files stay in `src/brain_sync/schemas/insights/` or
  move to a clearer package path?
  Default: keep the package location stable unless runtime loading becomes
  awkward.

- What exact CLI and doctor wording should be used for unsupported legacy
  brains?
  Default: instruct the user to re-initialize into a fresh Brain Format `1.0`
  brain and do not imply an automatic upgrade path exists.
