## Critical Findings

### 1. The operational event spine exists, but the in-code catalog is still fragmented

What is wrong:
Operational events are now emitted from a controlled set of files, but the
event families themselves still live as string literals spread across
`application/local_files.py`, `application/query_index.py`,
`application/sources.py`, `regen/engine.py`, `regen/queue.py`,
`runtime/repository.py`, `sync/finalization.py`, `sync/lifecycle.py`, and
`sync/reconcile.py`. The same family, `query.index.invalidated`, is emitted in
multiple modules through local helpers instead of one shared catalog.

Why it matters:
The architecture is ready for event-centralization work, but not finished.
Today the system relies on discipline rather than one code-level source of
truth for event family names and expected payload shape.

Evidence:

- `src/brain_sync/application/query_index.py`
- `src/brain_sync/regen/engine.py`
- `src/brain_sync/sync/lifecycle.py`
- `src/brain_sync/runtime/repository.py`
- `tests/unit/test_runtime_operational_event_callers.py`

Likely failure mode if left alone:
Future observability work will drift in naming, make event consumers brittle,
and weaken confidence in cross-flow diagnostics as the event set grows.

### 2. `regen/engine.py` is still the main internal REGEN blast radius

What is wrong:
The public REGEN boundary is now narrow, but `src/brain_sync/regen/engine.py`
still mixes folder classification, global-context cache handling, prompt
assembly, backend invocation, token telemetry, operational events, query-index
invalidation, and durable write coordination in one `1782`-line module.

Why it matters:
The next REGEN refactor is now architecturally local, which is real progress.
But that refactor still lands in a single dense internal owner, so cost,
quality, and performance work will remain high-diff unless the internals are
split further.

Evidence:

- `src/brain_sync/regen/__init__.py`
- `src/brain_sync/regen/engine.py`
- `src/brain_sync/regen/queue.py`
- `src/brain_sync/application/regen.py`

Likely failure mode if left alone:
Prompt-policy changes, cost/quality experiments, or backend-flow refactors
will keep touching one broad file that is hard to review and easy to
destabilize accidentally.

### 3. Sync source lifecycle ownership is truthful now, but concentrated in one oversized owner

What is wrong:
`src/brain_sync/sync/lifecycle.py` now owns source registration, update, move,
remove, missing observation, materialization commit ordering, child discovery,
watcher-driven folder handling, folder moves, and source reconcile. That makes
the ownership model much clearer than in earlier reviews, but it also creates
a new `1042`-line hotspot.

Why it matters:
This is the main “raised bar” change since review 3: lifecycle behavior is
much more coherent. The remaining risk is no longer cross-package confusion,
but internal concentration inside the sync owner that will grow again if the
next event and lifecycle changes keep landing there.

Evidence:

- `src/brain_sync/sync/lifecycle.py`
- `src/brain_sync/sync/finalization.py`
- `src/brain_sync/sync/lifecycle_policy.py`
- `docs/sync/README.md`
- `tests/integration/test_missing_source_commands.py`
- `tests/integration/test_source_finalization.py`

Likely failure mode if left alone:
Further lifecycle hardening, event catalog work, or source-admin behavior
changes will produce high-blast-radius edits inside one module rather than
smaller seam-local changes.

### 4. The live daemon path still falls back to coarse-grained reconcile and reload work

What is wrong:
`src/brain_sync/sync/daemon.py` still runs `reconcile_sources()` after watcher
event batches and still reloads full active source state on a fixed rescan
interval. `src/brain_sync/sync/lifecycle.py::reconcile_sources()` still walks
all manifests/runtime rows rather than a targeted subset.

Why it matters:
Correctness-first behavior is fine here, but the scale path remains broad even
after lifecycle ownership improved. Local churn in one area still pushes the
daemon back toward registry-wide work.

Evidence:

- `src/brain_sync/sync/daemon.py`
- `src/brain_sync/sync/lifecycle.py`
- `src/brain_sync/sync/source_state.py`
- `src/brain_sync/sync/reconcile.py`

Likely failure mode if left alone:
Brains with many sources or frequent local edits will keep paying repeated
whole-registry work, making performance improvements harder to localize later.

### 5. `runtime/repository.py` is a truthful boundary, but still too large for the next round of change

What is wrong:
`src/brain_sync/runtime/repository.py` still owns migrations, daemon guard and
status, polling rows, source lifecycle rows, child-discovery requests,
operational events, token telemetry, and regen locks in one `2329`-line module.

Why it matters:
This is much better than having multiple sqlite owners, and that improvement
should be preserved. The remaining issue is internal review and maintenance
cost inside the one truthful runtime owner.

Evidence:

- `src/brain_sync/runtime/repository.py`
- `tests/unit/test_architecture_boundaries.py`
- `tests/unit/test_runtime_operational_events.py`

Likely failure mode if left alone:
Event centralization, regen telemetry work, or future lifecycle changes will
keep adding to one already broad module and make safe refactoring slower.

## Verdict

Verdict: `Ready`

The sync source lifecycle area now clears a meaningfully higher bar than review
3. There is no remaining architecture blocker in that area that should stop
the next round of work.

Top 3 issues to manage during the next major refactor:

1. centralize operational event definitions without turning `runtime/` into a
   dumping ground for policy
2. split `regen/engine.py` internally while preserving the current narrow
   public REGEN surface
3. avoid growing `sync/lifecycle.py` further before extracting smaller
   lifecycle sub-owners

## Compatibility and Observability Readiness

Brain Format `1.2` safety:

Yes. The most likely next efforts, event-catalog centralization and REGEN
refactoring, are runtime/application/internal-structure changes. They do not
need portable manifest or managed-layout changes if the current ownership split
is preserved intentionally.

Observability readiness:

Mostly yes for structured logs and workflow-level operational timelines.
Partially for typed events and replay-oriented debugging: the event spine is
real, but it is still string-literal based and best-effort rather than backed
by a centralized catalog or stricter field contract. This is enough to justify
the next event-centralization effort, but not the end state.

The architecture is still not event-sourced. `operational_events` remain a
diagnostic trail layered over repository-owned state, not the system of record.

## Surgical Recommendations

1. Introduce one centralized operational event catalog, with shared event
   names and helper constructors for stable required fields.
2. Route all `query.index.invalidated` emission through one shared application
   or sync seam instead of duplicating the family in multiple modules.
3. Split `regen/engine.py` into smaller internal owners for prompt assembly,
   backend invocation/telemetry, and result-application/write coordination.
4. Split `sync/lifecycle.py` into smaller internal lifecycle services such as
   registration/admin flows, materialization commit, watcher/reconcile repair,
   and child-discovery follow-on handling.
5. Replace periodic full daemon source-state reload with an explicit runtime
   invalidation or registration-generation mechanism when practical.
6. Narrow watcher-triggered source reconcile toward targeted source/path repair
   instead of registry-wide sweeps where correctness allows it.
7. Add event-schema tests that assert the allowed event-family set and required
   fields per family, not just the current caller inventory.
