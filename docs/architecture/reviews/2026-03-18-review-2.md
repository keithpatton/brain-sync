## Critical Findings

### 1. REGEN still does not have a narrow public boundary

What is wrong:
`src/brain_sync/regen/__init__.py` still re-exports a wide mix of engine
internals, private-ish helpers, and queue mechanics. `src/brain_sync/regen/queue.py`
consumes `_PROPAGATES_UP`, `_parent_path`, `compute_waves`, `regen_path`, and
`regen_single_folder` through that barrel, while `src/brain_sync/sync/reconcile.py`
and `src/brain_sync/sync/watcher.py` still depend on tolerated `sync -> regen`
imports.

Why it matters:
The code is closer to a subsystem than it was at baseline, but the public
surface is still "most of the engine." That means the next REGEN refactor
would still be package-spanning rather than local.

Evidence:

- `src/brain_sync/regen/__init__.py`
- `src/brain_sync/regen/queue.py`
- `src/brain_sync/sync/reconcile.py`
- `src/brain_sync/sync/watcher.py`
- `tests/unit/test_architecture_boundaries.py`

Risk if left alone:
Engine decomposition or queue-policy changes will keep leaking across sync,
application, and tests, preserving the same broad blast radius the hardening
plan was trying to remove.

### 2. Runtime authority is clearer, but still not unified behind one truthful owner

What is wrong:
Cross-plane source and insight projection has moved into `application/`, but
machine-local persistence is still split between `runtime/repository.py`,
`runtime/child_requests.py`, and `runtime/token_tracking.py`. Both
`child_requests.py` and `token_tracking.py` talk to the runtime DB directly,
and `child_requests.py` reaches through the private `_connect()` helper while
duplicating table DDL already declared in `runtime/repository.py`.

Why it matters:
The baseline "runtime as dumping ground" problem is reduced, not eliminated.
The system now has better ownership boundaries conceptually than it has in its
runtime write APIs.

Evidence:

- `src/brain_sync/runtime/repository.py`
- `src/brain_sync/runtime/child_requests.py`
- `src/brain_sync/runtime/token_tracking.py`
- `src/brain_sync/application/source_state.py`
- `src/brain_sync/application/insights.py`

Risk if left alone:
Future runtime features will continue to choose between multiple persistence
styles, making migrations, instrumentation, and correctness checks harder to
standardize.

### 3. Sync still owns too much REGEN-adjacent policy inline

What is wrong:
`src/brain_sync/sync/daemon.py` now consumes cleaner application workflows, but
it still owns child-source placement rules, request clearing semantics,
structure-only insight-state repair, `_core` cache invalidation triggers, and
regen enqueue policy. `src/brain_sync/sync/watcher.py` still updates runtime
regen paths and portable manifests directly during folder moves.

Why it matters:
The transport-thinning work succeeded, but some of the policy moved sideways
into the daemon loop rather than behind smaller semantic services.

Evidence:

- `src/brain_sync/sync/daemon.py`
- `src/brain_sync/sync/watcher.py`
- `src/brain_sync/application/reconcile.py`
- `src/brain_sync/application/regen.py`
- `src/brain_sync/application/sources.py`

Risk if left alone:
Changes to child discovery, watcher behavior, or regen trigger semantics will
still require edits in the long-running loop and edge observers, increasing
regression risk across startup, live updates, and shutdown recovery.

### 4. The scale path still relies on whole-tree scans and repeated recomputation

What is wrong:
`AreaIndex.is_stale()` still rescans managed summaries under `knowledge/`,
startup reconcile still runs `classify_folder_change()` across every tracked
path, and the daemon still reloads full source state periodically and during
child-source registration.

Why it matters:
These are acceptable correctness-first choices for a moderate brain, but they
remain architectural cost centers rather than isolated implementation details.

Evidence:

- `src/brain_sync/query/area_index.py`
- `src/brain_sync/sync/reconcile.py`
- `src/brain_sync/sync/daemon.py`
- `docs/architecture/ARCHITECTURE.md`

Risk if left alone:
Larger brains will pay more at startup, on query freshness checks, and during
bursty change periods, making performance tuning harder to localize later.

### 5. Observability is still usage telemetry, not an operational event model

What is wrong:
The code records LLM invocation telemetry in `token_events`, but there is still
no typed event spine for source registration, reconcile outcomes, watcher move
handling, or regen lifecycle transitions at repository/application boundaries.

Why it matters:
The architecture is cleaner than the baseline, but it is still not
replay-friendly or correlation-friendly for structural debugging.

Evidence:

- `src/brain_sync/runtime/token_tracking.py`
- `src/brain_sync/application/reconcile.py`
- `src/brain_sync/application/sources.py`
- `src/brain_sync/sync/daemon.py`
- `docs/architecture/ARCHITECTURE.md`

Risk if left alone:
Production debugging will continue to depend on ad hoc logs and current-state
inspection rather than trustworthy timelines of what the system decided and why.

## Strengths

### 1. The child-discovery contract is now structurally truthful

`fetch_children` and `child_path` are no longer durable manifest fields.
Application source flows write runtime-only requests through
`runtime/child_requests.py`, manifests strip those fields on write, and tests
now assert the runtime-only behavior explicitly.

Evidence:

- `src/brain_sync/application/sources.py`
- `src/brain_sync/runtime/child_requests.py`
- `src/brain_sync/brain/manifest.py`
- `tests/integration/test_manifest_commands.py`

### 2. Transport-owned orchestration was meaningfully reduced

CLI and MCP now depend predominantly on `application/` workflows instead of
reaching directly into `brain`, `query`, `regen`, and runtime-reporting logic.
This is a real boundary improvement, not just a file move.

Evidence:

- `src/brain_sync/interfaces/cli/handlers.py`
- `src/brain_sync/interfaces/mcp/server.py`
- `src/brain_sync/application/__init__.py`
- `tests/unit/test_architecture_boundaries.py`

### 3. Single-brain runtime semantics are now explicit in both docs and code

The current architecture no longer pretends to be root-isolated at the runtime
layer. Active-root selection, status reporting, and runtime docs all describe
the same single-brain-per-config-directory model.

Evidence:

- `src/brain_sync/application/roots.py`
- `src/brain_sync/application/status.py`
- `src/brain_sync/runtime/config.py`
- `docs/architecture/ARCHITECTURE.md`
- `docs/runtime/SCHEMAS.md`

## Verdict

Verdict: `Not Yet`

Top 3 blocking issues before the next major REGEN refactor:

1. replace the broad `brain_sync.regen` barrel with a genuinely narrow public service boundary
2. finish runtime authority unification so machine-local persistence stops bypassing one consistent runtime API
3. shrink the remaining sync-owned REGEN policy and replace scan-heavy freshness paths with narrower invalidation seams

## Compatibility and Observability Readiness

Brain Format `1.0` safety:

Yes. The rerun shows that Brain Format `1.0` remains intentionally protected.
The portable contract is cleaner than the baseline because child-discovery is
now machine-local request state and the runtime layer is described explicitly
as single-brain for this stage. Compatibility tests still pass, including the
runtime child-request table shape and Brain Format `1.0` expectations.

Observability readiness:

Partially. The architecture is more ready for future instrumentation because
projection and transport ownership are cleaner, but it is still not ready for
typed events, replay-grade debugging, or event-sourced-style analysis. The
best future event seams are still repository and application boundaries:

- `brain/repository.py`
- `runtime/repository.py`
- `application/sources.py`
- `application/reconcile.py`
- `application/regen.py`
- `sync/daemon.py`

## Surgical Recommendations

1. Replace `brain_sync.regen` barrel exports with a small public API for queue, path regen, full regen, and folder classification.
2. Move queue-only helpers such as wave computation and propagation rules out of the public REGEN export surface.
3. Consolidate runtime DB access behind one runtime persistence surface so child requests and token telemetry stop opening their own DB paths directly.
4. Extract child-source placement and watcher-trigger policy from `sync/daemon.py` into smaller application- or sync-owned services with explicit inputs and outputs.
5. Introduce explicit invalidation/event hooks for summary writes, folder moves, and source registration changes so `AreaIndex` and startup reconcile can stop relying so heavily on tree scans.
6. Add typed operational events for source lifecycle, reconcile outcomes, watcher move handling, and regen ownership transitions.
7. Keep ratcheting the boundary tests by removing the remaining `sync -> regen` transitional debt entries once the new REGEN surface exists.
