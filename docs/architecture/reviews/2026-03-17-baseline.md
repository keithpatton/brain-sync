## Critical Findings

### 1. Child-discovery intent is structurally inconsistent

What is wrong:
`fetch_children` and `child_path` still exist as first-class fields on
`SourceManifest` in `src/brain_sync/brain/manifest.py`, are accepted by source
management in `src/brain_sync/application/sources.py`, and are merged back into
runtime state by `load_state()` in `src/brain_sync/runtime/repository.py`.
Writers then explicitly strip both fields during manifest serialization in
`src/brain_sync/brain/manifest.py`. The tests in
`tests/integration/test_manifest_commands.py` assert that this loss is
expected.

Why it matters:
This is not cosmetic debt. It means the code models durable source intent and
then intentionally discards it at the durable boundary while downstream code
continues to behave as if the intent exists.

Evidence:

- `src/brain_sync/brain/manifest.py`
- `src/brain_sync/runtime/repository.py`
- `tests/integration/test_manifest_commands.py`

Risk if unaddressed:
Future refactors will preserve the wrong abstraction. The system will keep
accepting one-shot child discovery as though it were a real contract while its
behavior remains process-local and brittle.

### 2. REGEN does not have a real public boundary yet

What is wrong:
`src/brain_sync/regen/__init__.py` re-exports internal helpers, private-ish
symbols, and engine details, and `src/brain_sync/regen/queue.py` consumes those
internals through the barrel export. `src/brain_sync/sync/reconcile.py` and
`src/brain_sync/sync/watcher.py` also import directly from `brain_sync.regen`
under an explicitly tolerated debt carveout in
`tests/unit/test_architecture_boundaries.py`.

Why it matters:
A major REGEN refactor cannot stay local if the subsystem boundary is just a
barrel file over engine internals.

Evidence:

- `src/brain_sync/regen/__init__.py`
- `src/brain_sync/regen/queue.py`
- `src/brain_sync/sync/reconcile.py`
- `src/brain_sync/sync/watcher.py`
- `tests/unit/test_architecture_boundaries.py`

Risk if unaddressed:
The REGEN refactor will punch through sync, watcher, queue, and transport code
instead of remaining a subsystem-local change.

### 3. Runtime ownership is still too monolithic to be a stable subsystem

What is wrong:
`src/brain_sync/runtime/repository.py` still mixes DB schema ownership, source
state projection, regen lifecycle state, daemon state, token-event tables,
portable sidecar projection, and retained compatibility stubs such as
`update_source_target_path()` and `update_source_flags()`.

Why it matters:
This file is acting as both a repository and a migration holding pen. That
keeps runtime as a dumping ground instead of a composable subsystem.

Evidence:

- `src/brain_sync/runtime/repository.py`
- tolerated debt entry in `tests/unit/test_architecture_boundaries.py`

Risk if unaddressed:
Any runtime or REGEN-adjacent change will keep landing in one high-blast-radius
module, increasing coupling and making correctness harder to verify.

### 4. Transports still own application policy directly

What is wrong:
`src/brain_sync/interfaces/mcp/server.py` directly imports `brain`, `query`,
and `regen` modules, builds and refreshes the `AreaIndex`, invokes `regen_all`
and `regen_path`, and uses `BrainRepository` directly. The CLI handler layer in
`src/brain_sync/interfaces/cli/handlers.py` also reaches directly into query,
regen, runtime-reporting, and reconcile logic.

Why it matters:
The documented model says `application/` owns interface-neutral workflows, but
the transports still embed important orchestration and policy.

Evidence:

- `src/brain_sync/interfaces/mcp/server.py`
- `src/brain_sync/interfaces/cli/handlers.py`
- `src/brain_sync/application/__init__.py`

Risk if unaddressed:
Policy changes, diagnostics, and refactors will be duplicated across CLI and
MCP surfaces instead of landing once in application-owned workflows.

### 5. Root scoping and runtime isolation are underdefined

What is wrong:
`resolve_root()` in `src/brain_sync/application/roots.py` simply chooses the
first configured brain. `runtime/token_tracking.py` accepts a `root` argument
but resolves the DB path through global runtime config, ignoring the root when
selecting storage.

Why it matters:
The current structure looks root-aware at call sites but remains effectively
global in runtime storage and root selection behavior.

Evidence:

- `src/brain_sync/application/roots.py`
- `src/brain_sync/runtime/token_tracking.py`

Risk if unaddressed:
Multi-brain support, per-brain observability, and correlation of operational
state will remain ambiguous and easy to corrupt.

## Strengths

### 1. The architecture contract is unusually explicit

`docs/RULES.md`, `docs/architecture/ARCHITECTURE.md`, and
`tests/unit/test_architecture_boundaries.py` form a real rules-plus-fitness
loop rather than vague architectural aspirations.

### 2. Brain Format `1.0` is materially protected

`docs/VERSIONING.md`, `docs/COMPATIBILITY.md`, and
`tests/compat/test_brain_format_v1.py` make the portable contract explicit and
verify that runtime rebuild is separate from durable brain state.

### 3. The source-adapter and test-tier story is strong

The `sources/` package remains mostly source-specific, and the test layout has
clear unit, integration, system, e2e, mcp, and compatibility roles instead of
one undifferentiated test bucket.

## Refactor Risk Map

Fragility points:

- `src/brain_sync/runtime/repository.py`
- `src/brain_sync/regen/engine.py`
- `src/brain_sync/interfaces/mcp/server.py`
- the child-discovery intent path spanning `application`, `brain`, `runtime`,
  and `sync`

Scaling risks:

- `AreaIndex.is_stale()` rescans managed summaries under `knowledge/`
- REGEN classification and content hashing still happen in paths that are also
  watcher and reconcile entrypoints
- transport-owned index lifecycle keeps query refresh tied to interface code

Likely REGEN-refactor breakpoints:

- `src/brain_sync/regen/__init__.py`
- `src/brain_sync/regen/queue.py`
- `src/brain_sync/sync/reconcile.py`
- `src/brain_sync/sync/watcher.py`
- `src/brain_sync/interfaces/mcp/server.py`

## Verdict

Verdict: `Not Yet`

Blockers before the next major REGEN refactor:

1. Resolve the child-discovery contract drift completely.
2. Replace the REGEN barrel export with a narrow public service boundary.
3. Split runtime ownership so REGEN and runtime changes stop defaulting to one
   monolithic repository module.

## Compatibility and Observability Readiness

Brain Format `1.0` safety:

The next hardening pass can and should preserve Brain Format `1.0`. Most of the
required cleanup is runtime, orchestration, and subsystem-boundary work rather
than a portable-brain format change. The main exception is child-discovery
intent: if it is meant to be durable, it would force a contract decision. The
better choice is to keep v1 frozen and make that intent runtime-only or remove
it.

Observability readiness:

The current structure is compatible with structured logs, but not yet with a
coherent event model or replay-grade debugging. Hooks should live at repository
and orchestration boundaries:

- `brain/repository.py`
- `runtime/repository.py`
- `sync/daemon.py`
- `sync/pipeline.py`
- `sync/reconcile.py`
- `regen` public service boundary once it exists

The current architecture is not event-sourced. It could support append-only
operational events, but that needs typed events, stable correlation ids, and a
clear owner for event emission before it becomes trustworthy.

## Surgical Recommendations

1. Delete the current split-brain handling of `fetch_children` and
   `child_path`: either model it as runtime-only intent with a single owner or
   remove it.
2. Replace `brain_sync.regen` barrel exports with a small public REGEN API and
   forbid sync and transport code from importing engine internals.
3. Break `runtime/repository.py` into focused owners for sync cache, regen
   locks, daemon status, and token events.
4. Move shared regen, query, and reconcile use cases into `application/` so
   CLI and MCP stop acting as parallel orchestration layers.
5. Introduce typed operational events at state and orchestration boundaries,
   with stable fields for session, owner, canonical id, knowledge path, outcome,
   and duration.
6. Replace whole-tree index staleness checks with invalidation signals emitted
   by summary writes, reconcile, and watcher-owned move handling.
7. Remove no-op compatibility stubs once callers are migrated so runtime APIs
   stop implying state transitions they no longer perform.
